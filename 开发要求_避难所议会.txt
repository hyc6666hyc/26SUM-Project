请为我完整实现一个可运行、可展示、结构清晰的期末设计项目：
.\.venv\Scripts\python.exe main.py
项目名称：
《避难所议会：基于大语言模型的多智能体生存与社会博弈模拟系统》

一、总体目标

实现一个回合制多人社会博弈游戏。游戏中多个玩家生活在资源有限的末日避难所中，需要处理突发事件、分配资源、讨论方案、交易、投票和执行行动。

玩家可以是：

1. AI Agent；
2. 真人玩家。

系统必须同时支持：

1. 纯 AI 自动对局；
2. 1 名真人与多个 AI Agent 混合对局；
3. 暂停、单步推进和自动推进；
4. 完整对局日志与赛后复盘。

游戏中包含普通阵营和隐藏破坏者。普通玩家既需要保证避难所存活，也需要完成自己的私人目标；隐藏破坏者需要暗中制造危机、误导决策并避免暴露。

请优先实现稳定、可运行、便于现场展示的 MVP，不要为了追求复杂而牺牲可运行性。

二、推荐技术栈

请使用：

- Python 3.11；
- Streamlit 作为前端；
- Pydantic 或 dataclass 定义数据模型；
- SQLite 或 JSON 保存对局状态、日志和记忆；
- 使用 .env 管理 API_KEY、BASE_URL、MODEL_NAME；
- pytest 编写核心规则测试。

本项目使用阿里云百炼 API。

通过 OpenAI Python SDK 调用百炼的 OpenAI 兼容接口。

环境变量：

DASHSCOPE_API_KEY
DASHSCOPE_BASE_URL
NORMAL_AGENT_MODEL
STRATEGY_AGENT_MODEL
REVIEW_MODEL

普通 Agent 默认使用 qwen3.6-flash。
破坏者、警卫等复杂决策角色默认使用 qwen3.7-plus。
赛后复盘默认使用 qwen3.7-plus。

所有模型调用必须经过统一 LLMClient 封装。
必须支持 JSON 结构化输出、超时、有限重试、异常处理和 fallback。
API 调用失败时，切换至规则型 Bot，不能让整局游戏卡死。
禁止在代码、日志、README 或 Git 仓库中保存真实 API Key。
请不要把游戏数值规则交给大语言模型处理。

规则引擎、行动合法性、资源变化、投票、胜负判断、秘密信息隔离和随机结果必须由确定性的 Python 代码负责。

LLM 只负责：

- 理解局势；
- 形成策略；
- 公开发言；
- 私聊；
- 提出方案；
- 选择投票；
- 选择合法行动；
- 更新简短记忆和玩家画像。

三、默认游戏配置

第一版固定为：

- 6 名玩家；
- 0 名或 1 名真人；
- 5 名普通玩家；
- 1 名隐藏破坏者；
- 共 6 个游戏日；
- 每名健康玩家每天 2 AP；
- 每名玩家每天最多公开发言 2 次；
- 每名玩家每天最多发起 1 次私聊；
- 每名玩家每天最多提出 1 个公共方案；
- 每名玩家每天最多执行 1 个秘密行动；
- 投票不消耗 AP；
- 游戏必须在有限步骤内结束。

需要允许在配置文件中修改：

- 玩家数量；
- 真人数量；
- 游戏天数；
- 是否启用破坏者；
- 随机种子；
- Agent 使用的模型；
- 自动推进速度。

四、世界状态

公共资源包含：

- food：食物；
- energy：能源；
- medicine：药品；
- parts：零件；
- stability：稳定度。

建议初始值：

- food = 40；
- energy = 35；
- medicine = 12；
- parts = 10；
- stability = 70。

核心设施包含：

- water_system：供水系统；
- power_system：发电系统；
- medical_system：医疗系统；
- communication_system：通信系统。

每个设施使用 0～100 的 durability 表示状态：

- 80～100：正常；
- 50～79：轻度损坏；
- 20～49：严重损坏；
- 0～19：失效。

设施状态必须真正影响资源消耗、事件后果和行动效果。

五、角色系统

至少实现以下普通职业：

1. 工程师 Engineer
   - 修理设施时只消耗 1 AP；
   - 修理效果高于其他职业；
   - 技能“紧急抢修”，冷却 3 天。

2. 医生 Doctor
   - 治疗时只消耗 1 AP；
   - 治疗效果高于其他职业；
   - 技能“全面诊疗”，冷却 3 天。

3. 警卫 Guard
   - 调查或保护时只消耗 1 AP；
   - 可以保护设施或玩家；
   - 技能“秘密监控”，冷却 2 天，可看到某玩家上一天执行的行动类型，但不能直接看到身份。

4. 商人 Trader
   - 每天可以额外发起一次交易；
   - 交易效率更高；
   - 技能“资源置换”，冷却 2 天。

5. 研究员 Researcher
   - 可以获得下一天事件的模糊提示；
   - 制造物资成本更低；
   - 技能“风险预测”，冷却 3 天。

6. 隐藏破坏者 Saboteur
   - 公开身份伪装成普通职业；
   - 拥有暗中破坏、偷窃资源、制造谣言、伪造记录等秘密能力；
   - 强力秘密行动通常消耗 2 AP；
   - 每天最多执行 1 个秘密行动；
   - 秘密行动可能留下不完整线索。

六、私人目标

每名普通玩家随机获得一个私人目标，例如：

工程师：
- 至少修复 2 次设施；
- 游戏结束时 energy >= 15。

医生：
- 至少 4 名玩家存活；
- 游戏结束时 medicine >= 3。

警卫：
- 成功阻止一次破坏；
- 参与驱逐真正的破坏者。

商人：
- 至少完成 3 次交易；
- 游戏结束时个人资源价值最高。

研究员：
- 成功分析 3 个事件；
- 制作至少 2 个特殊道具。

私人目标只对本人可见。

七、胜负条件

普通玩家完全胜利需要同时满足：

1. 避难所成功存活；
2. 完成自己的私人目标。

如果避难所存活，但私人目标未完成，则显示：

“生存成功，但个人目标未完成”。

避难所成功存活条件：

第 6 天结算后同时满足：

- stability > 0；
- food 和 energy 不同时为 0；
- 至少 3 名玩家仍在场；
- 至少 2 个核心设施没有完全失效。

避难所立即崩溃条件：

- stability <= 0；
- food 连续 2 天为 0；
- energy 连续 2 天为 0；
- 存活且未被驱逐的玩家少于 2 人；
- 所有核心设施失效。

隐藏破坏者满足以下任一条件即可获胜：

- 避难所崩溃；
- 第 6 天结束时 stability < 25；
- 完成指定的隐藏破坏目标。

但如果破坏者在第 4 天之前被成功驱逐，则不能获得“完全胜利”。

八、每日回合流程

每个游戏日严格分为：

1. EVENT：事件阶段；
2. DISCUSSION：讨论阶段；
3. ACTION：行动阶段；
4. VOTING：公共方案投票阶段；
5. RESOLUTION：结算阶段；
6. 可选 EXPULSION：驱逐阶段。

规则引擎必须阻止玩家在错误阶段执行动作。

1. 事件阶段
   - 随机生成或抽取当天事件；
   - 公布事件描述、公开后果和可选应对方式；
   - 玩家不能执行正式行动。

2. 讨论阶段
   - Agent 根据公共状态、私人目标、记忆和性格进行发言；
   - 可提出公共方案；
   - 可发起有限私聊或交易；
   - 不能无限讨论。

3. 行动阶段
   - 健康玩家获得 2 AP；
   - 可以执行一个 2 AP 动作；
   - 或两个 1 AP 动作；
   - 行动点不足时必须拒绝；
   - 非法行动必须返回明确原因。

4. 投票阶段
   - 对公共方案进行支持、反对或弃权；
   - 超过当前有效玩家半数支持才通过；
   - 平票或未过半则失败。

5. 结算阶段
   - 执行通过的方案；
   - 执行玩家公开行动；
   - 执行秘密行动；
   - 扣除每日资源；
   - 处理事件后果；
   - 更新设施、健康、稳定度和冷却；
   - 生成公共日志与私人日志；
   - 判断胜负。

九、基础动作

实现统一 Action 模型和合法性检查。

2 AP 动作：

- explore(area)：探索外部区域；
- repair(facility)：修理设施；
- investigate(target)：深度调查玩家、设施或事件；
- heal(player)：治疗重伤；
- craft(item)：制造物资；
- sabotage(target)：破坏者秘密破坏；
- steal(resource)：破坏者秘密偷窃。

1 AP 动作：

- search(location)：搜索少量资源或线索；
- transfer_resource(player, resource, amount)：转移资源；
- minor_heal(player)：简单治疗；
- reinforce(facility)：加固设施；
- rest()：休息；
- persuade(player)：拉拢或提出合作；
- spread_rumor(target)：破坏者制造谣言。

自由动作：

- 查看公共状态；
- 查看自己的私人状态；
- 投票；
- 接受或拒绝交易；
- 结束回合。

十、健康状态

至少实现：

- HEALTHY：2 AP；
- INJURED：1 AP；
- SERIOUSLY_INJURED：0 AP，但可有限发言和投票；
- UNCONSCIOUS：不能行动、发言或投票；
- ISOLATED：不能执行普通行动，但可交流和投票；
- EXPELLED：离场；
- DEAD：离场。

十一、公共方案

玩家可以提出公共方案。

Proposal 至少包含：

- id；
- proposer_id；
- title；
- description；
- resource_cost；
- participants；
- expected_effect；
- target_event；
- votes；
- status。

方案提交后不可随意修改。

方案必须由规则引擎检查：

- 资源是否足够；
- 参与者是否有效；
- 是否与当前事件相关；
- 是否重复提交；
- 是否超过每日提案限制。

十二、交易与承诺

交易流程：

1. 发起交易；
2. 指定 offer；
3. 指定 request；
4. 对方接受、拒绝或还价；
5. 规则引擎检查资源；
6. 执行交易；
7. 写入日志和记忆。

支持：

- 公开交易；
- 私下交易；
- 无条件赠送；
- 带承诺的交易。

承诺不由规则引擎强制履行，但必须记录。

Agent 对其他玩家维护：

- trust；
- suspicion；
- cooperation；
- honesty；
- usefulness；
- known_promises；
- broken_promises。

十三、秘密信息与权限隔离

必须严格区分：

公共信息：
- 当前天数；
- 公共资源；
- 设施状态；
- 当前事件；
- 公共聊天；
- 公开方案；
- 投票结果；
- 公开行动结果；
- 玩家是否在场。

私人信息：
- 私人目标；
- 隐藏阵营；
- 私聊；
- 私人资源；
- 个人线索；
- 秘密行动。

系统信息：
- 全部身份；
- 全部秘密行动；
- 完整事件参数；
- 随机结果；
- 真实胜负信息。

Agent 只能通过专用视图函数获取允许的信息。

不要把完整 GameState 直接传给任意 Agent。

十四、秘密行动与线索

秘密行动的公共日志不得显示执行者。

例如应显示：

“夜间，通信系统出现异常损坏，原因未知。”

而不是：

“四号破坏了通信系统。”

秘密行动可能生成不同强度线索：

- 弱线索：某人曾接近故障区域；
- 中等线索：发言与行动记录不一致；
- 强线索：持有被盗资源或破坏工具。

调查不能直接返回：

“某玩家就是破坏者。”

只能返回概率性、不完整、可能被伪造的线索。

十五、驱逐机制

必须实现驱逐破坏者或普通玩家的机制。

推荐规则：

1. 每两天最多发起一次驱逐；
2. 至少 2 名有效玩家联合提名同一人；
3. 被提名者获得一次公开辩护机会；
4. 其他玩家进行一次有限讨论；
5. 所有人秘密投票；
6. 超过当前有效玩家半数赞成才驱逐；
7. 平票则驱逐失败；
8. 被驱逐者失去行动、投票、发言和交易权限；
9. 被驱逐者可以进入旁观模式；
10. 真实身份在游戏结束前不公开；
11. 驱逐真正破坏者后，游戏继续；
12. 错误驱逐普通玩家时 stability -8；
13. 游戏结束后统一公开身份。

警卫的调查结果可以作为驱逐证据，但警卫的公开陈述仍可能是谎言，系统不要直接替玩家验证其说法。

十六、事件系统

至少准备 12 个事件，建议包括：

- 供水系统故障；
- 发电机过载；
- 食物仓库污染；
- 药品失踪；
- 新幸存者请求进入；
- 外部营地提出交易；
- 通信系统收到未知信号；
- 某玩家突然生病；
- 暴风雨即将到来；
- 地下区域出现异常声音；
- 仓库门锁被破坏；
- 内部出现资源争夺。

每个事件至少包含：

- id；
- title；
- description；
- visible_effect；
- available_solutions；
- resource_cost；
- success_effect；
- failure_effect；
- hidden_risk；
- related_facility；
- role_bonus。

事件需要有不同难度，且资源代价不能全部相同。

十七、Agent 设计

每个 Agent 至少包含：

- 公开职业；
- 隐藏阵营；
- 私人目标；
- 性格参数；
- 当前计划；
- 关系画像；
- 当前回合记忆；
- 整局关键记忆；
- 承诺记录；
- 交易记录；
- 投票记录；
- 行动记录。

性格参数建议：

- cooperation；
- risk_tolerance；
- selfishness；
- deception；
- suspicion；
- obedience；
- leadership；
- revenge。

Agent 每阶段必须输出结构化 JSON，不要依赖自由文本解析。

示例：

{
  "public_message": "我建议优先修复供水系统。",
  "private_messages": [],
  "proposal": null,
  "trade": null,
  "vote": "support",
  "actions": [
    {
      "type": "repair",
      "target": "water_system"
    }
  ],
  "memory_update": "三号承诺支持修理方案。",
  "relationship_updates": {
    "player_3": {
      "trust_delta": 5,
      "suspicion_delta": -2
    }
  }
}

必须使用 Pydantic 验证 LLM 输出。

如果输出格式错误：

1. 尝试一次修复；
2. 再失败则执行默认策略；
3. 不得卡死整个对局。

不要要求或保存模型的隐藏思维链。

只保存：

- 简短决策理由；
- 结构化行动；
- 可展示的策略摘要。

十九、记忆系统

MVP 使用 JSON 或 SQLite，不要一开始接入复杂向量数据库。

三层记忆：

1. 当前回合记忆；
2. 整局关键事件摘要；
3. 对其他玩家的关系画像。

每天结束时进行摘要压缩，避免 Prompt 无限增长。

记忆内容包括：

- 谁承诺了什么；
- 谁违约；
- 谁帮助过自己；
- 谁提出高风险方案；
- 谁发言与行动不一致；
- 谁可能是破坏者；
- 私人目标完成进度。

二十、随机性与可复现性

所有随机数必须由规则引擎生成。

必须支持 random_seed。

相同配置和相同 seed 应尽量得到一致的：

- 事件顺序；
- 探索结果；
- 初始角色；
- 私人目标；
- 调查线索；
- 随机伤害。

LLM 发言本身不要求完全一致。

- 驱逐流程；
- 游戏结束结果。

真人模式中，只显示该真人有权限看到的私人信息。

纯 AI 模式中可以提供“管理员观察模式”，但必须明确标记为调试视图。

二十二、自动对局

必须实现纯 AI 自动对局。

自动对局流程：

1. Agent 依次获取自己的合法视图；
2. Agent 进行有限发言；
3. Agent 提出方案；
4. Agent 进行投票；
5. Agent 选择行动；
6. 规则引擎结算；
7. 进入下一阶段或下一天；
8. 达到胜负条件时停止。

防止无限循环：

- 每个 Agent 每阶段最多决策一次；
- 发言、私聊、提案和工具调用均有限制；
- 设置 max_steps；
- 解析失败使用默认策略；
- 无合法行动时自动 rest 或 end_turn。

二十三、默认策略

普通 Agent 默认策略：

1. 优先避免立即崩溃；
2. 资源足够时支持低风险、高收益方案；
3. 受伤时优先治疗或休息；
4. 没有更好行动时搜索资源；
5. 根据私人目标选择次要优先级；
6. 投票优先公共生存。

破坏者默认策略：

1. 前期伪装成合作玩家；
2. 避免连续执行明显破坏；
3. 推动高成本、低收益方案；
4. 利用资源冲突制造分裂；
5. 后期再执行强力破坏；
6. 被高度怀疑时暂时帮助避难所。

二十四、日志与回放

需要记录：

- 每次状态变化；
- 每次发言；
- 私聊；
- 提案；
- 投票；
- 交易；
- 行动；
- 秘密行动；
- 调查线索；
- 驱逐；
- 胜负；
- LLM 调用异常。

日志分为：

- public_log；
- private_log；
- admin_log。

支持：

- 导出 JSON；
- 重新载入；
- 按天回放；
- 显示关键事件时间线。

二十五、赛后复盘

游戏结束后生成：

- 阵营胜负；
- 每名玩家个人胜负；
- 私人目标完成情况；
- 总积分；
- 最终身份；
- 关键发言；
- 关键投票；
- 关键交易；
- 关键破坏；
- 驱逐是否正确；
- 承诺与违约；
- 信任关系变化；
- 最有影响力玩家；
- 最可信玩家；
- 最可疑玩家；
- 每个 Agent 的简短策略总结。
